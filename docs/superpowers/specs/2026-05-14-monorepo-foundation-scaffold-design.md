## 🔍 Problem Summary

HireIQ needs a working monorepo scaffold that allows developers to begin building features without environment friction. At Milestone 1, there is no source code yet — only documentation and specs exist. The goal is to produce the minimum viable project structure that satisfies four concrete acceptance criteria:

1. **FastAPI** app runs on `localhost:8000` with a `/health` endpoint returning HTTP 200.
2. **SQLite** database initialises automatically on first run via a migration script.
3. **React/Vite** app runs on `localhost:5173` and proxies all `/api/*` requests to FastAPI (with prefix rewrite).
4. **README** documents complete setup steps for both frontend and backend.

The system must work with **zero external dependencies** (no Docker, no cloud services) at M1. Developers should be able to go from a fresh clone to a running full-stack app in under five minutes using `make` commands.

The existing [`ARCHITECTURE.md`](../../ARCHITECTURE.md) already prescribes the tech stack (FastAPI + SQLAlchemy + SQLite + React + Vite + GNU Make), module boundaries, directory structure, and key design patterns. This spec must remain strictly conformant with that document.

---

## 💡 Options Considered

The three meaningful design dimensions at M1 are: **(A) how the database auto-initialises**, **(B) how services start in parallel**, and **(C) how the developer workflow is orchestrated**. Three coherent option sets emerge:

---

### Option 1 — Explicit Three-Step Manual Workflow

**Summary:** Developer runs three separate, explicit commands: `make install` → `make migrate` → `make dev`. Each command does one thing. Nothing auto-runs anything else.

```bash
make install    # pip install + npm install
make migrate    # runs migrations/init_db.py → create_all
make dev        # starts uvicorn + vite in parallel
```

**Pros:**

- Maximum transparency — every step is visible and intentional
- Easiest to debug — if one step fails, it is clear which step failed
- Lowest Makefile complexity — three simple phony targets
- Aligns with the ARCHITECTURE.md database flow diagram exactly as written

**Cons:**

- ❌ Does **not** satisfy Acceptance Criteria #2 ("initialises *automatically* on first run") — requires developer to remember `make migrate` before `make dev`
- New developer friction — three commands instead of one
- `hireiq.db` will be missing if developer runs `make dev` without `make migrate` first, causing cryptic SQLite errors
- Violates the spirit of "without environment friction" from the user story

---

### Option 2 — FastAPI Lifespan Auto-Migration on App Startup

**Summary:** Database migration is triggered by FastAPI's startup lifecycle (`asynccontextmanager` lifespan). When uvicorn starts, the app calls `init_db()` automatically. `make dev` only needs to start both services in parallel.

```python
# app/main.py
from contextlib import asynccontextmanager
from app.migrations.init_db import run_migrations

@asynccontextmanager
async def lifespan(app: FastAPI):
    run_migrations()   # auto-init on every startup
    yield

app = FastAPI(lifespan=lifespan, ...)
```

**Pros:**

- True zero-friction auto-init — `make dev` is the only command needed after install
- Satisfies AC #2 exactly: SQLite initialises automatically on first server start
- Single source of truth — migration is always tied to app lifecycle

**Cons:**

- ❌ Couples application startup with database management — violates separation of concerns
- Makes `make migrate` redundant or confusing (it still exists but is not needed)
- Running migration on every startup introduces latency (acceptable at M1, but bad habit to establish)
- Harder to run migrations in CI/CD or production without starting the full server
- Diverges from the ARCHITECTURE.md flow diagram which explicitly shows `make migrate` as the entry point
- `init_db.py` must import `app.main` for model registration, creating a circular import risk

---

### Option 3 — `make dev` Depends on `migrate` Target ✅ Recommended

**Summary:** `make dev` declares `migrate` as a Makefile prerequisite. The first time `make dev` runs, migrations execute automatically before services start. `make migrate` remains a standalone, re-runnable target. Services launch in parallel using shell background processes with `trap`-based cleanup on exit.

```makefile
dev: migrate
	@trap 'kill %1 %2 2>/dev/null; exit 0' INT TERM; \
	cd backend && uvicorn app.main:app --reload --port 8000 & \
	cd frontend && npm run dev & \
	wait
```

**Pros:**

- ✅ Satisfies AC #2: database auto-inits before `make dev` starts services
- ✅ Migration logic stays in `migrations/init_db.py` — not mixed into app startup
- ✅ `make migrate` is still usable as a standalone command (CI, reset workflows)
- ✅ `create_all` is idempotent — re-running migration on every `make dev` is harmless
- ✅ Standard GNU Make dependency pattern — unsurprising to any developer
- ✅ `trap` ensures both background processes are killed on Ctrl+C (prevents port 8000 leak)
- Preserves ARCHITECTURE.md's existing flow diagrams without modification

**Cons:**

- Migration runs on every `make dev` invocation (acceptable: `create_all` is a no-op if tables exist)
- Parallel process management via `&` + `wait` is less ergonomic than `foreman`/`overmind` — but avoids adding a dependency
- Log output from both services interleaves in the terminal (expected behaviour, not a bug)

---

## ✅ Recommended Approach

**Option 3: `make dev` with `migrate` as a Makefile prerequisite.**

### Why Option 3 wins

| Criterion | Option 1 | Option 2 | **Option 3** |
|-----------|----------|----------|-------------|
| Satisfies AC #2 (auto-init) | ❌ | ✅ | **✅** |
| Respects separation of concerns | ✅ | ❌ | **✅** |
| Conforms to ARCHITECTURE.md | ✅ | ⚠️ partial | **✅** |
| Single command dev start | ❌ | ✅ | **✅** |
| `make migrate` standalone usable | ✅ | ⚠️ redundant | **✅** |
| No new runtime dependencies | ✅ | ✅ | **✅** |
| Graceful Ctrl+C cleanup | ❌ | ❌ | **✅** |

Option 1 fails the acceptance criteria. Option 2 couples concerns inappropriately and diverges from the architecture document. Option 3 satisfies all criteria cleanly, adds no dependencies, and uses standard patterns that every developer recognises.

**Step 1.2 — `backend/requirements.txt`**


**Step 1.3 — `backend/app/database.py`**

```python
import os
from sqlalchemy import create_engine
from sqlalchemy.orm import sessionmaker, declarative_base
from dotenv import load_dotenv

load_dotenv()

DATABASE_URL = os.getenv("DATABASE_URL", "sqlite:///./hireiq.db")

engine = create_engine(
    DATABASE_URL,
    connect_args={"check_same_thread": False},  # required for SQLite + FastAPI threading
)

SessionLocal = sessionmaker(autocommit=False, autoflush=False, bind=engine)

Base = declarative_base()
```

> **Note:** `check_same_thread=False` is mandatory. FastAPI's threaded execution model will otherwise raise `SQLite objects created in a thread can only be used in that same thread`.

**Step 1.4 — `backend/app/routers/health.py`**

```python
from datetime import datetime, timezone
from fastapi import APIRouter

router = APIRouter()

@router.get("/health", status_code=200)
def health_check() -> dict:
    """Returns service liveness status and current UTC timestamp."""
    return {
        "status": "ok",
        "timestamp": datetime.now(timezone.utc).isoformat(),
    }
```

> **Key:** Route is `/health`, not `/api/health`. The Vite proxy rewrites `/api/health` → `/health` before it reaches FastAPI. Defining `/api/health` in FastAPI would make the effective proxied URL `/api/api/health`.

**Step 1.5 — `backend/app/main.py`**

```python
from fastapi import FastAPI
from fastapi.middleware.cors import CORSMiddleware
from app.routers import health

app = FastAPI(title="HireIQ API", version="0.1.0")

app.add_middleware(
    CORSMiddleware,
    allow_origins=["http://localhost:5173"],  # explicit — never "*"
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)

app.include_router(health.router)
```

**Step 1.6 — `backend/migrations/init_db.py`**

```python
"""Idempotent database initializer. Safe to run multiple times."""
import sys
import os

sys.path.insert(0, os.path.dirname(os.path.dirname(os.path.abspath(__file__))))

from app.database import engine, Base
import app.main  # noqa: F401 — triggers model registration with Base

def run():
    print("Running migrations: Base.metadata.create_all()")
    Base.metadata.create_all(bind=engine)
    print("Migrations complete.")

if __name__ == "__main__":
    run()
```

**Step 1.7 — `backend/.env.example`**

```bash
# Copy to .env and adjust for your environment
# DATABASE_URL defaults to SQLite file at backend/hireiq.db
DATABASE_URL=sqlite:///./hireiq.db
```

**Step 1.8 — Backend tests: `backend/tests/test_health.py`**

```python
from fastapi.testclient import TestClient
from app.main import app

client = TestClient(app)

def test_health_returns_200():
    response = client.get("/health")
    assert response.status_code == 200

def test_health_returns_status_ok():
    response = client.get("/health")
    data = response.json()
    assert data["status"] == "ok"

def test_health_returns_timestamp():
    response = client.get("/health")
    data = response.json()
    assert "timestamp" in data
    assert data["timestamp"]  # non-empty string
```

**Step 2.1 — Bootstrap Vite + React + TypeScript**

**Step 2.2 — `frontend/vite.config.ts`**

```typescript
import react from '@vitejs/plugin-react'

export default defineConfig({
  plugins: [react()],
  server: {
    port: 5173,
    proxy: {
      '/api': {
        target: 'http://localhost:8000',
        changeOrigin: true,
        rewrite: (path) => path.replace(/^\/api/, ''),
      },
    },
  },
})
```

**Step 2.3 — `frontend/src/App.tsx`** (minimal health display)

```tsx
import { useState, useEffect } from 'react'

type HealthStatus = {
  status: string
  timestamp: string
} | null

function App() {
  const [health, setHealth] = useState<HealthStatus>(null)
  const [error, setError] = useState<string | null>(null)

  useEffect(() => {
    fetch('/api/health')
      .then((res) => {
        if (!res.ok) throw new Error(`HTTP ${res.status}`)
        return res.json() as Promise<HealthStatus>
      })
      .then(setHealth)
      .catch((err: Error) => setError(err.message))
  }, [])

  if (error) return <div>Backend error: {error}</div>
  if (!health) return <div>Connecting to backend...</div>
  return (
    <div>
      <h1>HireIQ</h1>
      <p>Backend status: <strong>{health.status}</strong></p>
      <p>Last check: {health.timestamp}</p>
    </div>
  )
}

export default App
```

### Phase 3 — Root Makefile Orchestration

**Step 3.1 — Root `Makefile`**

```makefile
.DEFAULT_GOAL := help
.PHONY: install dev migrate lint test help

## install: Install all backend and frontend dependencies
install:
	@echo "→ Installing backend dependencies..."
	cd backend && pip install -r requirements.txt
	@echo "→ Installing frontend dependencies..."
	cd frontend && npm install
	@echo "✓ All dependencies installed."

## migrate: Run database migrations (idempotent — safe to re-run)
migrate:
	@echo "→ Running database migrations..."
	cd backend && python migrations/init_db.py
	@echo "✓ Migrations complete."

## dev: Start backend and frontend development servers (auto-runs migrate first)
dev: migrate
	@echo "→ Starting backend on http://localhost:8000"
	@echo "→ Starting frontend on http://localhost:5173"
	@echo "   Press Ctrl+C to stop both servers."
	@trap 'kill %1 %2 2>/dev/null; exit 0' INT TERM; \
	cd backend && uvicorn app.main:app --reload --port 8000 & \
	cd frontend && npm run dev & \
	wait

## lint: Run linters on backend (ruff) and frontend (tsc)
lint:
	@echo "→ Linting backend..."
	cd backend && ruff check .
	@echo "→ Type-checking frontend..."
	cd frontend && npx tsc --noEmit

## test: Run all tests
test:
	@echo "→ Running backend tests..."
	cd backend && pytest tests/ -v

## help: Show available make targets
help:
	@grep -E '^## [a-zA-Z_-]+:' Makefile | sed 's/## /  make /'
```

## ⚠️ Risks / Edge Cases

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| **Port 8000 leak after Ctrl+C** | High | Medium | `trap 'kill %1 %2' INT TERM` in Makefile `dev` target. Document `kill $(lsof -ti:8000)` in README troubleshooting. |
| **`hireiq.db` resolves to wrong directory** | Medium | High | Makefile always `cd backend &&` before running uvicorn. SQLite URL `sqlite:///./hireiq.db` resolves relative to CWD. CWD must be `backend/`. |
| **`init_db.py` misses new models** | Medium | Medium | `init_db.py` imports `app.main` which triggers all router imports. New model files must be imported in their router module. Document in AGENTS.md gotchas. |
| **Double `/api` prefix in FastAPI routes** | Medium | High | FastAPI routes are `/health`, NOT `/api/health`. Vite proxy rewrites `/api/health` → `/health`. Already documented in AGENTS.md gotchas #2. |
| **`check_same_thread=False` missing** | Low | High | Included in `database.py` spec above. Without it, FastAPI threading causes runtime SQLite errors. |
| **Windows compatibility** | Medium | Medium | `trap` + `kill` require bash/sh. Windows users must use WSL or Git Bash. Documented in README prerequisites. |
| **TypeScript errors silent in dev mode** | Low | Medium | Vite dev server tolerates TS errors; production build fails on them. `make lint` runs `tsc --noEmit`. Must run before PR. |
| **CORS wildcard (`allow_origins=["*"]`)** | Low | High (security) | Spec explicitly uses `["http://localhost:5173"]`. CONSTITUTION.md prohibits `"*"`. Enforced at code review. |

## 📌 Assumptions

1. **Python 3.10+** is available on developer machines.
2. **Node.js 18 LTS** is available on developer machines.
3. **GNU Make** is available — standard on macOS and Linux; Windows developers use WSL or Git Bash.
4. **No virtual environment tooling mandated** — `pip install -r requirements.txt` targets whatever Python environment the developer chooses. `venv/` and `.venv/` are gitignored.
5. **SQLite is the only database at M1** — no PostgreSQL, no Redis, no external services.
6. **No authentication at M1** — `/health` is a public, unauthenticated endpoint.
7. **No Alembic at M1** — `Base.metadata.create_all()` is the migration strategy. Alembic will be introduced at M2 when the schema begins to evolve incrementally.
8. **`make dev` is the canonical developer workflow** — it is the documented entry point; running uvicorn or `npm run dev` manually is fine but not documented.
9. **React App.tsx displays health status only** — M1 frontend exists to prove the Vite proxy works end-to-end, not to implement hiring features.
10. **Ruff is the Python linter** — `ruff check .` runs from `backend/`. Not flake8 or pylint.
