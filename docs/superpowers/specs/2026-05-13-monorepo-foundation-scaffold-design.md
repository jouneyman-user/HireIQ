## 🔎 Problem Summary

The HireIQ repository is a clean slate — a single stub `README.md` on the `main` branch (initial commit `90bb3b0`). Milestone 1 requires bootstrapping a **working monorepo scaffold** so that every subsequent feature can be developed without environment friction.

The target stack is:

| Layer | Technology | URL |
|-------|-----------|-----|
| Backend API | FastAPI (Python) | `http://localhost:8000` |
| Database | SQLite (auto-init on first run) | `backend/hireiq.db` |
| Frontend | React + TypeScript (Vite) | `http://localhost:5173` |

**Core acceptance criteria:**

1. `GET /health` → HTTP 200 on the FastAPI server.
2. SQLite initialises automatically via a migration script on first run.
3. React app proxies all `/api/*` calls to FastAPI — zero CORS config needed in the browser.
4. `README.md` documents setup steps for both backend and frontend.

The effort is rated **Small**, which constrains the approach to lean tooling with minimal prerequisites.

## 💡 Options Considered

### Option 1 — Minimal Flat Monorepo + Makefile Orchestration ✅ *(Recommended)*

Two sibling directories (`backend/`, `frontend/`) at the repo root with a `Makefile` that wires both together. No container runtime required.

**Pros:**
- Zero extra tooling prerequisites beyond Python ≥ 3.10 and Node ≥ 18
- `make install && make migrate && make dev` satisfies all four acceptance criteria in one line
- Fast iteration loop — no image builds, instant hot-reload on both sides
- `make` is universally understood and cross-platform via `winget`/`choco` on Windows
- SQLite file lives alongside backend code — trivial path resolution
- Aligns perfectly with "S" effort rating

**Cons:**
- Local Python/Node versions must match manually (no lockfile enforcement at the OS level)
- No built-in environment parity guarantee across machines (mitigated by documenting prerequisites clearly)

---

### Option 2 — Docker Compose Dev Environment

Backend and frontend each containerised; a `docker-compose.yml` at the repo root orchestrates both services.

**Pros:**
- Consistent, reproducible environment across all developer machines
- Easy to add services (Redis, PostgreSQL) in future milestones without changing dev workflow
- No local Python/Node installation required

**Cons:**
- Requires Docker Desktop — significant friction for a foundation milestone where the goal is *zero friction*
- SQLite volume mounts introduce edge cases (file locking, permissions on Linux vs macOS vs Windows)
- Slower hot-reload cycle inside containers (bind mount overhead)
- Overkill for "S" effort rating; deferred to a future "productionisation" milestone

---

### Option 3 — Poetry + PNPM Workspaces + `concurrently`

Full monorepo workspace tooling: Poetry for Python dependency management, PNPM workspaces for JavaScript packages, `concurrently` for a unified dev process from the repo root.

**Pros:**
- Modern, strict lockfiles on both sides (`poetry.lock`, `pnpm-lock.yaml`)
- Scales cleanly as the project grows into multiple frontend packages or Python services
- Single `pnpm run dev` from root starts everything

**Cons:**
- PNPM workspace config is unnecessary complexity when there is only one frontend package
- Poetry is non-standard for developers unfamiliar with it; adds a learning curve
- More setup steps directly contradicts the acceptance criterion "without environment friction"
- PNPM is not universally installed; adds a new prerequisite on top of Node

---

## ✅ Recommended Approach

**Option 1 — Minimal Flat Monorepo + Makefile.**

The acceptance criteria explicitly value *friction-free setup*. The effort is rated Small. Docker adds prerequisites and complexity; workspace tooling is premature with only one backend and one frontend. A flat `backend/` + `frontend/` layout with a `Makefile` achieves all four acceptance criteria with the least moving parts and can be evolved toward Docker Compose or workspace tooling in a later milestone when environment parity becomes a genuine problem.

### 2. Backend — `backend/`

#### `app/main.py`
```python
from fastapi import FastAPI
from fastapi.middleware.cors import CORSMiddleware
from app.routers import health

app = FastAPI(title="HireIQ API", version="0.1.0")

app.add_middleware(
    CORSMiddleware,
    allow_origins=["http://localhost:5173"],
    allow_methods=["*"],
    allow_headers=["*"],
)

app.include_router(health.router)
```

#### `app/database.py`
```python
from sqlalchemy import create_engine
from sqlalchemy.ext.declarative import declarative_base
from sqlalchemy.orm import sessionmaker
import os

DATABASE_URL = os.getenv("DATABASE_URL", "sqlite:///./hireiq.db")

engine = create_engine(DATABASE_URL, connect_args={"check_same_thread": False})
SessionLocal = sessionmaker(autocommit=False, autoflush=False, bind=engine)
Base = declarative_base()
```

#### `app/routers/health.py`
```python
from fastapi import APIRouter
from datetime import datetime, timezone

router = APIRouter()

@router.get("/health", status_code=200)
def health_check():
    return {"status": "ok", "timestamp": datetime.now(timezone.utc).isoformat()}
```

#### `migrations/init_db.py`
```python
from app.database import Base, engine
import app.main  # noqa: F401 — triggers model imports so Base knows about all tables

if __name__ == "__main__":
    Base.metadata.create_all(bind=engine)
    print("Database initialised.")
```

Idempotent — safe to run multiple times; existing tables are not dropped.

#### `requirements.txt`


**Start command:** `uvicorn app.main:app --reload --port 8000` (run from `backend/`)

```tsx
import { useEffect, useState } from 'react'

function App() {
  const [status, setStatus] = useState<string>('loading…')

  useEffect(() => {
    fetch('/api/health')
      .then((r) => r.json())
      .then((d) => setStatus(d.status))
      .catch(() => setStatus('unreachable'))
  }, [])

  return (
    <div>
      <h1>HireIQ</h1>
      <p>API status: <strong>{status}</strong></p>
    </div>
  )
}

export default App
```

```makefile
dev:            ## Start backend and frontend concurrently
	cd backend && uvicorn app.main:app --reload --port 8000 &
	cd frontend && npm run dev

	cd frontend && npm run lint
```

## ⚠️ Risks / Edge Cases

| Risk | Mitigation |
|------|-----------|
| Port 8000 or 5173 already in use | Document `BACKEND_PORT` / `VITE_PORT` env var overrides in README |
| Python version mismatch (`<3.10`) | Document prerequisite clearly; add `python_requires >= "3.10"` note in README |
| SQLite file permission errors | Store `hireiq.db` inside `backend/` (not repo root); add to `.gitignore` |
| CORS errors during development | Explicitly whitelist `http://localhost:5173` in `CORSMiddleware` |
| `make` not available on Windows | Document Git Bash and alternative manual commands |
| Vite proxy rewrite strips `/api` prefix | `rewrite` rule in `vite.config.ts` removes `/api` so FastAPI routes stay clean (e.g. `/health` not `/api/health`) |
| `init_db.py` run before models are imported | Import `app.main` in `init_db.py` to trigger all model registrations before `create_all` |
| Alembic vs. raw `create_all` | `Base.metadata.create_all` is sufficient and simpler for M1; migrate to Alembic when the first schema change is needed in M2+ |
| `uvicorn` background process leak in `make dev` | Document `Ctrl+C` stops the foreground Vite process; kill background uvicorn with `kill $(lsof -ti:8000)` |

## 📌 Assumptions

1. **Python ≥ 3.10** and **Node.js ≥ 18** are available in all developer environments.
2. The project name *HireIQ* implies a hiring-intelligence platform; no domain models (candidates, jobs, etc.) are needed at M1 — only the infrastructure skeleton.
3. **TypeScript** is used for the React frontend (standard Vite `react-ts` template provides superior DX from day one).
4. **SQLite** is the sole database for all environments in M1; a migration to PostgreSQL is a future milestone concern.
5. No authentication, CI/CD pipeline, or deployment configuration is in scope for M1.
6. The `/health` endpoint is a **public liveness probe** — no authentication required.
7. `hireiq.db` is git-ignored; the `init_db.py` migration script is the sole source of truth for schema initialisation.
8. CORS is configured permissively for `localhost` origins only in development; production hardening (e.g., specific domain allow-list) is deferred to a deployment milestone.
9. The `make dev` target uses a simple background `&` pattern rather than `concurrently` to avoid adding an extra Node dependency at M1; this can be upgraded if a more polished dev experience is desired.
10. No virtualenv management tool (venv, pyenv, conda) is prescribed — developers use their preferred Python environment manager.
