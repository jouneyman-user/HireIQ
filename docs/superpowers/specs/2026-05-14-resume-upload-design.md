> **Issue:** [#2 — Resume Upload — Upload and store candidate resumes](https://github.com/jouneyman-user/HireIQ/issues/2)

> **Effort:** M (Medium)
> **Status:** Approved for implementation
> **Date:** 2026-05-14
> **Generated by:** `implementation-plan` skill

---

## 🔍 Problem Summary

HireIQ's M1 foundation scaffold is complete. The platform now needs its first domain feature: **candidate resume upload**. Hiring teams need a way to submit candidate resumes (PDF or DOCX) into the system so they can be stored, listed, and later processed by AI features in M3+.

At M2, the scope is deliberately narrow:

1. **Backend**: A `POST /resumes` endpoint that accepts a multipart file upload, validates it, stores it on the local filesystem, and persists metadata (candidate name, email, filename, content type, upload timestamp) to SQLite.
2. **Backend**: `GET /resumes` (list) and `GET /resumes/{id}` (retrieve metadata) endpoints.
3. **Database**: A `Resume` SQLAlchemy model added to the existing `Base` — migrations remain `Base.metadata.create_all` (no Alembic yet).
4. **Frontend**: A React file-upload form component with drag-and-drop support, file-type validation, upload progress indicator, and a list view of uploaded resumes.
5. **Local storage only**: Uploaded files are written to `backend/uploads/resumes/` — no cloud storage, no S3, no CDN at M2.

This feature must remain **strictly conformant** with the patterns established in `ARCHITECTURE.md` and `CONSTITUTION.md`: router-per-domain, session-per-request, idempotent migrations, no wildcard CORS, no secrets in source, and zero new required external services.

---

## 💡 Options Considered

Three meaningful design dimensions exist for M2: **(A) file storage strategy**, **(B) resume text extraction**, and **(C) database model scope**. Two coherent option sets per dimension are evaluated below.

---

### Dimension A — File Storage Strategy

#### Option A1 — Local Filesystem Storage (UUID-named files) ✅ Recommended

Files are stored at `backend/uploads/resumes/<uuid>.<ext>`. The UUID is generated server-side on upload; the original filename is preserved only as a metadata field in the database row.

```
backend/
└── uploads/
    └── resumes/
        ├── 3f2e1a9b-8c74-4d01-a5e6-12abc3d45e67.pdf
        └── 7a1b2c3d-4e5f-6a7b-8c9d-0e1f2a3b4c5d.docx
```

**Pros:**
- Zero external dependencies — consistent with M1's local-only philosophy
- UUID filenames eliminate filename collisions without needing a deduplication strategy
- `uploads/` is git-ignored (same pattern as `hireiq.db`)
- Directory creation is idempotent (`os.makedirs(..., exist_ok=True)`)

**Cons:**
- Files lost if `backend/uploads/` is deleted (acceptable: M2 is local dev only)
- Not suitable for multi-instance deployment (deferred: M2 is single-machine)

#### Option A2 — MinIO / S3-Compatible Object Storage

Store files in a local MinIO container, accessed via `boto3`/`aioboto3`.

**Pros:**
- Production-parity storage from day one
- Separates file storage lifecycle from application lifecycle

**Cons:**
- Requires a running MinIO container — breaks the "zero external services" constraint from CONSTITUTION.md
- Adds `boto3` + MinIO client to `requirements.txt` — significant new surface
- Overkill for M2 scope and "M" effort rating

**Decision:** Option A1. Zero-dependency local storage is the right choice at M2.

---

### Dimension B — Resume Text Extraction

#### Option B1 — No extraction at M2 ✅ Recommended

Store the raw file only. No text extraction, no parsing, no AI inference.

**Pros:**
- Keeps M2 scope tight — upload/store/list is the acceptance criteria, not parsing
- Avoids adding `pdfminer`, `python-docx`, or `unstructured` to requirements
- Extraction is a natural M3 feature tied to AI inference

**Cons:**
- Resume content is not searchable at M2 (acceptable: no search feature exists yet)

#### Option B2 — Basic text extraction on upload (pdfminer + python-docx)

Extract raw text at upload time and store it in a `text_content` column.

**Pros:**
- Text is immediately available for future M3 AI features
- Avoids a backfill migration later

**Cons:**
- `pdfminer.six` and `python-docx` add ~30 MB of new dependencies
- Extraction failures would block uploads — poor UX for a foundation feature
- Text quality from PDFs is highly variable; cleaning required before AI use
- Scope creep: M2 acceptance criteria don't mention extraction

**Decision:** Option B1. No extraction at M2. Text extraction is an explicit M3 concern.

---

### Dimension C — Database Model Scope

#### Option C1 — Minimal Resume model (file metadata only) ✅ Recommended

The `Resume` table captures only what is needed to track an uploaded file:

```python
class Resume(Base):
    __tablename__ = "resumes"

    id         = Column(Integer, primary_key=True, index=True)
    candidate_name  = Column(String(255), nullable=False)
    candidate_email = Column(String(255), nullable=False, index=True)
    original_filename = Column(String(512), nullable=False)
    stored_filename   = Column(String(512), nullable=False, unique=True)  # UUID-based
    content_type      = Column(String(100), nullable=False)
    file_size_bytes   = Column(Integer, nullable=False)
    uploaded_at       = Column(DateTime, default=datetime.utcnow, nullable=False)
```

**Pros:**
- Minimal surface — easy to extend incrementally
- No foreign keys at M2 (no `Candidate` or `Job` entity yet — those are M3+)
- All fields are immediately useful (listing, filtering by email, auditing)

**Cons:**
- `candidate_name` and `candidate_email` are denormalized (will be normalized in M3 when a `Candidate` table is introduced)

#### Option C2 — Full Candidate + Resume relational model

Introduce a `Candidate` table now and foreign-key `Resume` to it.

**Pros:**
- Correct relational design from the start

**Cons:**
- `Candidate` entity requirements are undefined at M2 — premature normalization
- More migration complexity when `Candidate` requirements are clarified in M3
- Significantly increases M2 scope beyond "M" effort rating

**Decision:** Option C1. Minimal model; denormalization is acceptable until M3 defines `Candidate` properly.

---

## ✅ Recommended Approach

**Local filesystem storage (A1) + No extraction (B1) + Minimal Resume model (C1).**

| Criterion | A1 (local FS) | A2 (MinIO) | Winner |

### Phase 2 — Backend: Upload Service & Router

**Step 2.1 — `backend/app/routers/resumes.py`**

```python
import os
import uuid
from datetime import datetime
from fastapi import APIRouter, UploadFile, File, Form, HTTPException, Depends
from sqlalchemy.orm import Session
from app.database import SessionLocal
from app.models.resume import Resume

UPLOAD_DIR = "uploads/resumes"
ALLOWED_TYPES = {"application/pdf", "application/vnd.openxmlformats-officedocument.wordprocessingml.document"}
MAX_FILE_BYTES = 10 * 1024 * 1024  # 10 MB

router = APIRouter(prefix="/resumes", tags=["resumes"])

def get_db():
    db = SessionLocal()
    try:
        yield db
    finally:
        db.close()

@router.post("/", status_code=201)
async def upload_resume(
    candidate_name: str = Form(...),
    candidate_email: str = Form(...),
    file: UploadFile = File(...),
    db: Session = Depends(get_db),
):
    if file.content_type not in ALLOWED_TYPES:
        raise HTTPException(status_code=415, detail="Only PDF and DOCX files are accepted.")

    contents = await file.read()
    if len(contents) > MAX_FILE_BYTES:
        raise HTTPException(status_code=413, detail="File exceeds 10 MB limit.")

    ext = os.path.splitext(file.filename or "")[1].lower() or ".bin"
    stored_name = f"{uuid.uuid4()}{ext}"
    dest = os.path.join(UPLOAD_DIR, stored_name)

    with open(dest, "wb") as f:
        f.write(contents)

    resume = Resume(
        candidate_name=candidate_name,
        candidate_email=candidate_email,
        original_filename=file.filename or stored_name,
        stored_filename=stored_name,
        content_type=file.content_type,
        file_size_bytes=len(contents),
        uploaded_at=datetime.utcnow(),
    )
    db.add(resume)
    db.commit()
    db.refresh(resume)
    return resume

@router.get("/")
def list_resumes(db: Session = Depends(get_db)):
    return db.query(Resume).order_by(Resume.uploaded_at.desc()).all()

@router.get("/{resume_id}")
def get_resume(resume_id: int, db: Session = Depends(get_db)):
    resume = db.query(Resume).filter(Resume.id == resume_id).first()
    if not resume:
        raise HTTPException(status_code=404, detail="Resume not found.")
    return resume
```

### Phase 3 — Backend: Tests

**Step 3.1 — `backend/tests/test_resumes.py`**

```python
import io
import pytest
from fastapi.testclient import TestClient
from app.main import app
from app.database import Base, engine

client = TestClient(app)

@pytest.fixture(autouse=True)
def reset_db(tmp_path, monkeypatch):
    """Run against an in-memory SQLite DB; redirect uploads to tmp_path."""
    import app.routers.resumes as r
    monkeypatch.setattr(r, "UPLOAD_DIR", str(tmp_path))
    Base.metadata.create_all(bind=engine)

def _pdf_file():
    return ("file", ("test.pdf", io.BytesIO(b"%PDF-1.4 fake content"), "application/pdf"))

def test_upload_returns_201():
    response = client.post("/resumes/", data={"candidate_name": "Alice", "candidate_email": "alice@test.com"}, files=[_pdf_file()])
    assert response.status_code == 201

def test_upload_returns_resume_metadata():
    response = client.post("/resumes/", data={"candidate_name": "Alice", "candidate_email": "alice@test.com"}, files=[_pdf_file()])
    data = response.json()
    assert data["candidate_name"] == "Alice"
    assert data["candidate_email"] == "alice@test.com"
    assert data["content_type"] == "application/pdf"
    assert "id" in data

def test_upload_rejects_invalid_type():
    bad_file = ("file", ("test.txt", io.BytesIO(b"hello"), "text/plain"))
    response = client.post("/resumes/", data={"candidate_name": "Bob", "candidate_email": "bob@test.com"}, files=[bad_file])
    assert response.status_code == 415

def test_list_resumes():
    client.post("/resumes/", data={"candidate_name": "Alice", "candidate_email": "alice@test.com"}, files=[_pdf_file()])
    response = client.get("/resumes/")
    assert response.status_code == 200
    assert len(response.json()) >= 1

def test_get_resume_by_id():
    upload = client.post("/resumes/", data={"candidate_name": "Alice", "candidate_email": "alice@test.com"}, files=[_pdf_file()])
    resume_id = upload.json()["id"]
    response = client.get(f"/resumes/{resume_id}")
    assert response.status_code == 200
    assert response.json()["id"] == resume_id

def test_get_resume_404():
    response = client.get("/resumes/99999")
    assert response.status_code == 404
```

### Phase 4 — Frontend: Resume Upload Component

**Step 4.1 — `frontend/src/components/ResumeUpload.tsx`**

```tsx
import { useState, useRef } from 'react'

type Resume = {
  id: number
  candidate_name: string
  candidate_email: string
  original_filename: string
  content_type: string
  file_size_bytes: number
  uploaded_at: string
}

type UploadState = 'idle' | 'uploading' | 'success' | 'error'

export function ResumeUpload({ onUploaded }: { onUploaded: (r: Resume) => void }) {
  const [name, setName] = useState('')
  const [email, setEmail] = useState('')
  const [file, setFile] = useState<File | null>(null)
  const [state, setState] = useState<UploadState>('idle')
  const [error, setError] = useState<string | null>(null)
  const [dragOver, setDragOver] = useState(false)
  const inputRef = useRef<HTMLInputElement>(null)

  const ACCEPTED = ['application/pdf', 'application/vnd.openxmlformats-officedocument.wordprocessingml.document']

  const handleFile = (f: File) => {
    if (!ACCEPTED.includes(f.type)) {
      setError('Only PDF and DOCX files are accepted.')
      return
    }
    if (f.size > 10 * 1024 * 1024) {
      setError('File must be under 10 MB.')
      return
    }
    setError(null)
    setFile(f)
  }

  const handleDrop = (e: React.DragEvent) => {
    e.preventDefault()
    setDragOver(false)
    const dropped = e.dataTransfer.files[0]
    if (dropped) handleFile(dropped)
  }

  const handleSubmit = async (e: React.FormEvent) => {
    e.preventDefault()
    if (!file || !name || !email) return
    setState('uploading')
    try {
      const form = new FormData()
      form.append('candidate_name', name)
      form.append('candidate_email', email)
      form.append('file', file)
      const res = await fetch('/api/resumes/', { method: 'POST', body: form })
      if (!res.ok) {
        const body = await res.json()
        throw new Error(body.detail ?? `HTTP ${res.status}`)
      }
      const uploaded: Resume = await res.json()
      setState('success')
      onUploaded(uploaded)
      setName(''); setEmail(''); setFile(null)
      setTimeout(() => setState('idle'), 3000)
    } catch (err: unknown) {
      setError(err instanceof Error ? err.message : 'Upload failed.')
      setState('error')
    }
  }

  return (
    <form onSubmit={handleSubmit} style={{ maxWidth: 480 }}>
      <h2>Upload Resume</h2>
      <input placeholder="Candidate name" value={name} onChange={e => setName(e.target.value)} required style={{ display: 'block', width: '100%', marginBottom: 8 }} />
      <input type="email" placeholder="Candidate email" value={email} onChange={e => setEmail(e.target.value)} required style={{ display: 'block', width: '100%', marginBottom: 8 }} />
      <div
        onDragOver={e => { e.preventDefault(); setDragOver(true) }}
        onDragLeave={() => setDragOver(false)}
        onDrop={handleDrop}
        onClick={() => inputRef.current?.click()}
        style={{ border: `2px dashed ${dragOver ? '#0070f3' : '#ccc'}`, borderRadius: 8, padding: 24, textAlign: 'center', cursor: 'pointer', marginBottom: 8 }}
      >
        {file ? file.name : 'Drop PDF or DOCX here, or click to browse'}
        <input ref={inputRef} type="file" accept=".pdf,.docx" style={{ display: 'none' }} onChange={e => { if (e.target.files?.[0]) handleFile(e.target.files[0]) }} />
      </div>
      {error && <p style={{ color: 'red' }}>{error}</p>}
      {state === 'success' && <p style={{ color: 'green' }}>Resume uploaded successfully!</p>}
      <button type="submit" disabled={state === 'uploading' || !file || !name || !email}>
        {state === 'uploading' ? 'Uploading...' : 'Upload Resume'}
      </button>
    </form>
  )
}
```

**Step 4.2 — `frontend/src/components/ResumeList.tsx`**

```tsx
type Resume = {
  id: number
  candidate_name: string
  candidate_email: string
  original_filename: string
  file_size_bytes: number
  uploaded_at: string
}

export function ResumeList({ resumes }: { resumes: Resume[] }) {
  if (resumes.length === 0) return <p>No resumes uploaded yet.</p>
  return (
    <table style={{ width: '100%', borderCollapse: 'collapse', marginTop: 24 }}>
      <thead>
        <tr>
          {['Name', 'Email', 'File', 'Size', 'Uploaded'].map(h => (
            <th key={h} style={{ textAlign: 'left', borderBottom: '1px solid #ccc', padding: '6px 8px' }}>{h}</th>
          ))}
        </tr>
      </thead>
      <tbody>
        {resumes.map(r => (
          <tr key={r.id}>
            <td style={{ padding: '6px 8px' }}>{r.candidate_name}</td>
            <td style={{ padding: '6px 8px' }}>{r.candidate_email}</td>
            <td style={{ padding: '6px 8px' }}>{r.original_filename}</td>
            <td style={{ padding: '6px 8px' }}>{(r.file_size_bytes / 1024).toFixed(1)} KB</td>
            <td style={{ padding: '6px 8px' }}>{new Date(r.uploaded_at).toLocaleString()}</td>
          </tr>
        ))}
      </tbody>
    </table>
  )
}
```

## ⚠️ Risks / Edge Cases

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| **`uploads/resumes/` missing at startup** | Medium | High | `os.makedirs(..., exist_ok=True)` in `init_db.py`; router also creates on startup |
| **MIME type spoofing (client sends wrong content-type)** | Medium | Medium | Validate `content_type` against allowlist; M3+ can add magic-byte checking |
| **File size limit bypass via chunked upload** | Low | Medium | Read entire file into memory for size check; acceptable at M2 scale |
| **UUID collision (astronomically unlikely)** | Very Low | Low | `unique=True` on `stored_filename` column; DB will reject duplicate |
| **`python-multipart` missing from requirements** | High (if forgotten) | High | Explicitly add to `requirements.txt`; FastAPI will error on startup without it |
| **Concurrent uploads writing to same path** | Very Low | Low | UUID filenames eliminate collision; SQLite WAL mode handles concurrent writes |
| **`.gitignore` missing `uploads/`** | Medium | Low | Add `backend/uploads/` to `.gitignore` in migration phase |
| **DOCX detection via content-type** | Medium | Medium | DOCX content-type is verbose — document exact string in code comment |
| **Frontend form submits without file** | Low | Low | `disabled` attribute on submit button enforces presence |
| **Test isolation: shared `uploads/` directory** | Medium | Medium | `monkeypatch.setattr(r, "UPLOAD_DIR", str(tmp_path))` redirects uploads per test |

## 📌 Assumptions

1. **M1 is complete and merged**: FastAPI health endpoint running, SQLite database initializing, Vite proxy configured.
2. **No authentication at M2**: Resume upload is an unauthenticated endpoint — authentication is deferred.
3. **Local filesystem only**: No S3, MinIO, or CDN. Files stored at `backend/uploads/resumes/`.
4. **PDF and DOCX only**: No RTF, TXT, HTML, or image-format resumes accepted.
5. **10 MB per file**: Sufficient for all standard resume formats; large portfolios are out of scope.
6. **No text extraction**: Raw file storage only; AI parsing is an M3 feature.
7. **Denormalized candidate fields**: `candidate_name` and `candidate_email` live on the `Resume` model; a separate `Candidate` entity is an M3 concern.
8. **Idempotent migrations**: `Base.metadata.create_all` remains the migration strategy; adding `Resume` to the model registry is sufficient.
9. **`python-multipart` is a runtime dependency**: Required by FastAPI for `UploadFile` / `Form` parsing.
10. **Frontend uses plain `fetch` API**: No Axios, no React Query, no SWR — consistent with M1's minimal dependency philosophy.
11. **Vitest is added for frontend unit tests** in M2; component-level tests for `ResumeUpload` are in scope.

## Definition of Done

All items must be true before M2 is closed:

- [ ] `POST /resumes/` accepts PDF and DOCX; rejects all other types with HTTP 415
- [ ] `POST /resumes/` enforces 10 MB file size limit; returns HTTP 413 on exceed
- [ ] Uploaded file is stored at `backend/uploads/resumes/<uuid>.<ext>`
- [ ] Resume metadata is persisted to `resumes` table in SQLite
- [ ] `GET /resumes/` returns all resumes ordered by upload time (newest first)
- [ ] `GET /resumes/{id}` returns a single resume or HTTP 404
- [ ] `make migrate` creates the `resumes` table and `uploads/resumes/` directory
- [ ] `make test` passes: all backend tests green (upload, list, get, reject-invalid-type, 404)
- [ ] `make lint` passes: no ruff errors, no TypeScript errors
- [ ] Frontend `ResumeUpload` component accepts drag-and-drop and click-to-browse
- [ ] Frontend displays upload success / error feedback
- [ ] Frontend `ResumeList` displays all uploaded resumes in a table
- [ ] `backend/uploads/` is in `.gitignore`
- [ ] `python-multipart` is in `requirements.txt`
- [ ] CORS `allow_origins` remains `["http://localhost:5173"]` — not `["*"]`
